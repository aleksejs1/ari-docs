# Relationship Playbooks

Relationship Playbooks is a goal-driven task scheduling system that attaches a concrete action plan to any contact. Instead of just reminding you to reach out, a Playbook tells you *what to do* â€” phone calls, date nights, surprises, check-in questions â€” and spaces them out at a frequency that fits your goal for that relationship.

---

## Core Concepts

| Concept | Description |
|---|---|
| **Playbook** | A goal + a series of recurring tasks attached to a single contact |
| **Goal** | The intent behind the relationship (Maintain, Deepen, Reignite, Rekindle, Appreciate) |
| **Preset** | A named task template (e.g. *Monthly Phone Call*, *Quarterly Reconnect*) that matches the goal |
| **Task** | A single actionable item generated by the Playbook (e.g. "Phone Call", "Date Night") |
| **Reflection** | An optional note you write after completing a task |

## Working with Tasks

The **Playbook** card on a contact's detail page shows up to 5 active tasks. Each task has:

| Action | Description |
|---|---|
| **Mark done** | Completes the task. Online tasks are marked immediately; offline tasks (calls, visits) enter a short reflection window first. |
| **Snooze** | Delays the task by a few days without losing it. |
| **Skip** | Archives the task and moves on. |
| **Reflect** | Opens a reflection prompt after an offline task. |

### Task Types

| Type | Description |
|---|---|
| Phone Call | Call the contact |
| Video Call | Hop on a video call |
| Visit in Person | Meet face to face |
| Send a Text | Write a short message |
| Date Night | Plan an outing together |
| Do a Surprise | Do something unexpected for them |
| Shared Activity | Do something together (workout, movie, etc.) |
| Share an Insight | Send them an article, idea, or recommendation |
| Check-in Question | Ask a meaningful question to deepen the conversation |
| Restart Message | Re-initiate contact after a gap |

## Offline Tasks & Reflections

Tasks marked `is_offline=true` (phone calls, visits, date nights) move to an **Awaiting Reflection** state when you mark them done. A prompt appears asking *"How did it go?"* â€” a context-specific question for each task type.

- If you fill in the reflection, it is saved immediately.
- If you skip or close without answering, the system automatically finalises the task after **48 hours** via a background job.

---

## Pause & Resume

Click the **Pause** icon in the Playbook card header to pause the Playbook. While paused:
- No new tasks are generated.
- Existing pending tasks are set to `paused` status and hidden from the card.
- A **Paused** badge appears in the header.

Click the **Resume** icon to resume. Pending tasks are restored with recalculated due dates.

---

## Switching or Removing a Playbook

- **Switch** â€” Click the **+** icon to open the Wizard again with a new goal/preset. Active tasks from the old plan are archived.
- **Remove** â€” Click the trash icon. All pending tasks are archived and the Playbook is deleted.

## Celebration Milestone

Every 4 tasks completed in a series triggers a **celebration screen** that acknowledges your progress. Dismiss it to continue.

---

## Reciprocity Indicator

Below the task list, a small stat shows the interaction balance over the last 90 days:

> *Over the last 90 days: you reached out 5 times, they reached out 2 times.*

This is drawn from the interaction log (field: **Initiator**) and gives you a quick read on whether the relationship is one-sided.

---

## Seasonal Check-in Banner

Once per season (every 90 days), a banner appears at the top of the Dashboard for users who have at least one active Playbook:

> **Time for a seasonal check-in!**
> A new season is here. Reflect on your relationships and set intentions.

Clicking **Dismiss** marks the notification as read and hides the banner.

## Cron Jobs

The following background commands power Relationship Playbooks:

| Command | Schedule | Description |
|---|---|---|
| `playbooks:generate-tasks` | Daily | Generates the next task for each active Playbook whose task queue is empty |
| `playbooks:finalise-reflections` | Hourly | Finalises tasks stuck in `awaiting_reflection` for more than 48 hours |
| `playbooks:seasonal-checkin` | Quarterly | Creates seasonal check-in notifications for users with active Playbooks |

## Developer Notes

### Key Entities

| Entity | Description |
|---|---|
| `ContactPlaybook` | Stores the active plan for a contact: `preset`, `goal`, `whyTags`, `whyText`, `status`, and `celebrationPending`. One-to-one with a contact (a contact may have at most one non-archived playbook at a time). |
| `ContactTask` | A single actionable item belonging to a `ContactPlaybook`. Tracks `type`, `seriesKey`, `status`, `dueDate`, `isOffline`, `snoozedUntil`, `reflectionDueAt`, and `completedAt`. The `seriesKey` (defaults to `type`) identifies the recurring series within a preset and is used by `ContactTaskGeneratorService` to find the most recent task per series when computing the next due date. |
| `TaskReflection` | A OneToOne entity attached to a `ContactTask`. Created by `TaskReflectionFactory` when an offline task is marked done. Holds a `question` (context-specific per task type, max 500 chars) and the user's `answer` (max 10 000 chars). Fetched with `fetch: EAGER` on the `ContactTask` mapping to avoid an N+1 query when the task list is serialized. |
| `PlaybookTemplate` | A configuration-only entity (or YAML/PHP config resource) that defines the available preset names, their goal, and the task type mix. Not a Doctrine entity â€” presets are resolved by name at activation time. |

### Celebration Milestone

`ContactPlaybook::CELEBRATION_MILESTONE = 4`. Every time a task in a given series is completed and the per-series completion count is a multiple of 4, `ContactPlaybook::celebrationPending` is set to `true`. The API response for the next GET on the playbook will include `celebrationPending: true`. The frontend shows the celebration screen and then PATCHes `celebrationPending=false` to reset it.

### TaskReflection and N+1 Prevention

`ContactTask::$reflection` is declared with `fetch: 'EAGER'` on the `#[ORM\OneToOne]` mapping. This means Doctrine joins the `task_reflection` table in the same query when loading a `ContactTask`, avoiding one extra query per task when serializing the task list. The `reflection` field is included in the `contact_task:read` normalization group, so it is always returned inline in task API responses.

### Rate Limiting

Playbook activation (`POST /api/contacts/{contactId}/playbook`) is rate-limited by `JwtUserRateLimiterSubscriber` to **20 activations per hour** per authenticated user. Requests exceeding the limit receive HTTP `429 Too Many Requests`. This covers switching playbooks (which archives the old one and activates a new one) as well as fresh activations.
